# ROS 2 Humble Lab 4: Launch Files, Rosbag, and rqt-plot

## Project Context

**Objectives:** Automating node startup with launch files, recording/replaying topic data with Rosbag, and visualizing velocity via rqt_plot.

**Key Tasks:** 
- Modified a launch file to spawn two turtles simultaneously using topic remapping.
- Implemented a 'Follow-the-Leader' Python node where Turtle 2 subscribes to the pose of Turtle 1 to track it.

---

## Approach

To achieve the project objectives, the following technical steps and logic were implemented:

1. **Launch File Configuration:** 
   A custom Python launch file (`two_turtles_launch.py`) was created to automate the initialization process. Instead of manually running the `turtlesim_node` multiple times, the launch file uses the `Node` action from `launch_ros` to start two distinct turtlesim windows. Topic remapping was utilized to ensure that the two turtles operated in different namespaces (e.g., `/turtlesim1` and `/turtlesim2`), preventing topic conflicts while allowing them to run concurrently.

2. **Follow-the-Leader Logic:** 
   A custom ROS 2 Python node (`follower.py`) was developed to implement the tracking behavior. The logic relies on subscribing to Turtle 1's pose (`/turtle1/pose`) and publishing velocity commands (`/turtle2/cmd_vel`) to Turtle 2. 
   - The node calculates the Euclidean distance and the angular difference between Turtle 1 and Turtle 2.
   - Using a simple proportional controller (P-controller), it computes linear and angular velocities to minimize these errors, driving Turtle 2 toward Turtle 1.
   - The velocity commands are continuously published in real-time, resulting in a smooth following behavior.

---

## Findings

1. **Rosbag Data Recording (`/turtle1/pose`):** 
   Using the `ros2 bag record` tool, we successfully captured the pose data of Turtle 1 as it was manually driven around the environment. Replaying this bag file with `ros2 bag play` perfectly reproduced Turtle 1's movements, demonstrating that Rosbag is an incredibly effective tool for logging state data and testing nodes without needing a human to manually provide inputs repeatedly.

2. **Velocity Visualization via `rqt_plot`:** 
   When analyzing the velocity topics using `rqt_plot`, the graphs revealed distinct characteristics of manual control. Manual inputs (via the `teleop_turtle` node) resulted in sudden spikes and step-changes in linear and angular velocities. The graphs visually confirmed the discrete nature of keyboard control, as opposed to the smooth, continuous velocity curves generated by the proportional controller of the follower node.

---

## Observations

1. **Efficiency of Launch Files:**
   Utilizing launch files drastically improved workflow efficiency. Booting up the entire simulation—including multiple nodes, remapped namespaces, and specific parameters—took only a single terminal command (`ros2 launch`). This is a significant improvement over the tedious and error-prone process of opening multiple terminal tabs, sourcing the workspace, and running individual nodes sequentially.

2. **Stability of the Tracking Behavior:**
   The proportional tracking behavior exhibited good stability overall. Turtle 2 successfully minimized the distance to Turtle 1. However, some minor oscillations in the angular velocity were observed when Turtle 1 made sharp turns or when the turtles got too close to each other. Fine-tuning the proportional gain constants ($K_p$) could help smooth out these aggressive corrective movements and enhance tracking precision.
